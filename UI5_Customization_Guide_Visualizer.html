Unit 01 · Setup

Start Here & Run the App

Before changing anything, get the app running so you can see every edit live. You'll work entirely inside one folder — webapp/ — and refresh the browser after each save.

Step 0 — Run it with Mock Data

Open a terminal in the project folder and run:

# starts the app with fake (mock) data — no real server needed npm run start-mock

This opens the app in your browser. Keep it running — every time you save a file, refresh the browser to see the change.

🏅
Golden rule for a beginner: text goes in i18n, not in the XML. In the views you'll see things like text="{i18n>welcomeHeadline}" — that {i18n>…} means "look up this word in the i18n file." So to change wording, you edit i18n — not the screen.

Change one thing, then refresh. Never make five edits at once — if the app breaks, you won't know which edit caused it. Small steps are the fastest way to learn.
Q: You want to rename the app title. Which file do you edit — the view XML or i18n?
✓ i18n (webapp/i18n/i18n.properties). The view only shows {i18n>appTitle} — a pointer. The actual word lives in i18n, so changing it there updates every screen that uses that key.

🔗
The link that beginners break: the event name in the view (press=".onContactPress") must exactly match the function name in the controller (onContactPress). A typo here is the #1 cause of "is not a function" errors.
Q: Your new button shows but clicking it does nothing / throws "is not a function". Why?
✓ The press=".onContactPress" in the view doesn't match a function of the same name in the controller — either it's misspelled or the function is missing. Names are case-sensitive and must match exactly.

Unit 06 · The #1 Concept — Interactive

Reading Bindings {…}

Everywhere in the views you see curly braces. That's binding — the UI pulls values from a data source instead of hard-coded text. Learn to read the model name before the > and you'll know exactly which file to edit. Click each example.

{i18n>welcomeHeadline}
{view>Product/Name}
{Name}
{= ${appView>/layout}... }
🔑
The rule: the word before > is the model name. {i18n>…} → the i18n file. {view>…} → the view model. No prefix (like {Name}) → the default model, which is your product/OData data. Once you know the model, you know the file.
Q: You see {Name} with no model prefix. Where does that value come from?
✓ The default model — your product data (from Products.json in mock mode, or OData in a real system). No prefix always means the default model.
Unit 07 · Troubleshooting — Interactive

Debugging & Common Errors

When a change breaks the app, don't panic — the browser tells you what's wrong. Press F12 → Console and read the red error. Click each error below to see the real cause.

…is not a function
▸ tap to see the cause & fix
Cause: a typo in a press=".onXxx" name, or the function is missing in the controller. Fix: make the view's event name match the controller function exactly (case-sensitive).
Cannot read ... of undefined
▸ tap to see the cause & fix
Cause: a binding path points to data that doesn't exist. Fix: check the path in {…} — the field name may be misspelled or missing from the model/JSON.
XML parse error / app won't load
▸ tap to see the cause & fix
Cause: an unclosed tag or a missing / in a .view.xml. Fix: every tag must close — <Button …/> or <Button>…</Button>.
XML Rules That Prevent 90% of Breakage

Attributes need quotestext="Hello"text=Hello
Names are case-sensitiveButton, pressbutton, Press
Event ↔ function matchview & controller identicalany spelling mismatch
🔍
Best debugging tool: install the "UI5 Inspector" browser extension (or press Ctrl+Shift+Alt+S for the built-in Diagnostics). Click any element on screen to see its control type, properties, and which binding feeds it — it answers "which file controls this thing?" instantly.
Q: The app shows a blank screen after your edit. First thing to do?
✓ Press F12 → Console and read the red error. It names the problem and often the file/line. A blank screen is usually an XML parse error (unclosed tag) or a bad binding path.
Unit 08 · Best Practice — Interactive

Golden Safety Habits

Tick these off as habits. They prevent almost every "I broke it and don't know how" moment. Tap each to check it.

✓
Back up before editing. Copy a file (e.g. Cart.view.xml → Cart.view.xml.bak) so you can restore it. Even better, run git init to undo cleanly.
✓
Change one thing, then refresh. Never make five edits at once — if it breaks, you won't know which one did it.
✓
Never edit node_modules/. That's the framework. Only edit inside webapp/.
✓
Keep i18n keys, change only values. Deleting a key a view uses shows the raw key on screen (e.g. welcomeHeadline) instead of text.
✓
Match the existing style. Copy the pattern of a neighbouring function or tag — consistent code means fewer bugs and easier help later.

Unit 09 · Review

Quick-Fire Revision

Five questions to lock it in. Answer out loud before revealing.

Q1: To change wording, which file — view or i18n? Why?
✓ i18n. The view only holds {i18n>key} pointers; the actual text lives in i18n.properties, shared by every screen.
Q2: What are the three files you edit to add a new button?
✓ i18n.properties (the text) → *.view.xml (the control) → *.controller.js (the click handler). Text → view → controller.
Q3: In {i18n>title}, what does the word before the > tell you?
✓ The model name — i.e. which data source (and therefore which file) the value comes from. No prefix = the default (product) model.
Q4: The app broke after an edit. What's your first move?
✓ Open F12 → Console and read the red error — it tells you what and often where. Then undo your last single change.
Q5: Where do mock products live, and can you edit them safely?
✓ webapp/localService/mockdata/Products.json — yes, editing names/prices there is safe and needs no code.
🎯
One-line summary: stay inside webapp/, change one thing at a time, put text in i18n, read the model name in {bindings} to find the right file, add elements with the text → view → controller trio, and let the F12 console guide you when something breaks.
