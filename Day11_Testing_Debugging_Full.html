Unit 01 · Foundation

Testing vs Debugging

Testing checks whether an application behaves according to the business requirement. Debugging is locating, understanding and fixing a defect that testing found. One asks "does it work?", the other asks "why doesn't it?"

✅ Testing verifies…
The list loads · search & filter work · orders open · mandatory fields validate · Save sends correct data · success/error messages show · unauthorized users are blocked.
🔍 Debugging investigates…
Which line failed · what a variable actually held · whether the event fired · whether the request was sent · what the server returned · where reality diverged from intent.
🚗
Real-world analogy — a car factory: before delivery the company tests that the engine starts, brakes work, lights turn on, steering responds. When a brake fails, the engineer debugs — checking fluid, cable, pedal, pads, control unit — until the faulty part is found. Testing proves the whole car; debugging hunts one fault.
The Testing & Debugging Lifecycle
Testing finds it · debugging explains it · fixing removes it · retesting confirms it.
Why It Matters — a Small Defect, a Big Problem
🧮
An invoice app should compute Total = Quantity × Price. If a typo makes it Quantity + Price, then for Quantity 10 and Price ₹500 the correct ₹5,000 becomes a wrong ₹510. A single QUnit test on that function catches it before production.
Q: In the car analogy, what is the equivalent of a QUnit unit test?
✓ Checking one brake sensor in isolation. QUnit tests one small piece of logic on its own — like verifying a single component works before the whole car (the full app / OPA5 journey) is driven through a test route.
Unit 02 · Strategy

The Testing Pyramid

SAP Fiori uses four kinds of testing. The pyramid shows the healthy balance — many small fast tests at the base, fewer slow broad ones on top. Click each layer to explore.

🌐 End-to-End / Manual
few · slowest · whole business workflow
🎭 OPA5 — Integration
some · medium · parts working together in the UI
🧩 QUnit — Unit Tests
many · fastest · one small piece of code
📱
Analogy — a phone factory: before assembling the whole phone, each component is tested — battery, camera, speaker, display, microphone (QUnit). Then the assembled phone is switched on and used (OPA5). Then it's shipped and used on a real network (end-to-end).
The Fourth Type — Regression Testing
🔁
Regression testing ensures a new change hasn't broken existing features. Change the discount calculation, and regression tests re-confirm normal pricing, tax, existing-order display, and saving all still work. In practice, your existing QUnit + OPA5 suite is your regression net — re-run it after every change.
⚖️
Why a pyramid, not a rectangle? Lower levels have more tests because they're faster and target smaller code. Flipping it (mostly slow UI/manual tests) is the classic "ice-cream cone" anti-pattern — slow and brittle.
Q: Which layer should contain the most tests, and why?
✓ The base — unit tests (QUnit). They're fast and focus on small sections of code, so you write many. Fewer OPA5 integration tests sit above, and only a handful of end-to-end/manual tests at the very top.

Unit 04 · Running for Testing

Mock Data vs Real Backend

You can run a Fiori app two ways: against the real OData backend, or against a local mock server that returns fake data. Each answers different questions — and the run command comes straight from package.json.

The Run Commands
📦 package.json scripts
"scripts": { "start": "fiori run --open index.html", "start-mock": "fiori run --config ui5-mock.yaml --open index.html" } # npm start → real backend # npm run start-mock → local mock data
🗄️ Real Backend
App → Destination → OData Service → SAP Backend → Business Data. Tests real data, authorization, connectivity, business logic, and real error responses.
🧪 Mock Server
App → Local Mock Server → metadata.xml + JSON. Test the frontend with predictable data, without depending on an available backend.
Where Test Files Live — Project Structure
🗂️ project layout webapp/
webapp/ ├── controller/ # Main.controller.js ├── view/ # Main.view.xml ├── model/ # formatter.js ← unit-tested logic ├── localService/ │ ├── metadata.xml │ └── mockdata/# Products.json ← fake data ├── test/ │ ├── unit/ # QUnit tests (model/formatter.js) │ └── integration/ # OPA5 journeys + pages/ ├── Component.js └── manifest.json
📄
Example mock data (Products.json) holds records like { ProductID:"P1001", ProductName:"Laptop", Price:65000, Stock:12 }. With these you can test table binding, search, filtering, navigation, and formatters — no real SAP system needed.
Q: You want to test your table binding and search on a plane with no VPN to SAP. Which command?
✓ npm run start-mock. The mock server serves local JSON via ui5-mock.yaml, so the frontend runs on predictable fake data with no backend connection required.

The Code Under the Breakpoint
📁 Main.controller.js webapp/controller/Main.controller.js
onCalculateTotal: function () { const oModel = this.getView().getModel("order"); const iQuantity = oModel.getProperty("/Quantity"); const fPrice = oModel.getProperty("/Price"); const fTotal = iQuantity * fPrice; // ⛔ breakpoint here oModel.setProperty("/Total", fTotal); }

📹
Analogy — pausing security footage. Normally the video runs non-stop; a breakpoint is the Pause button. Frozen, you inspect who entered, what they carried, which door they used — just as a breakpoint lets you inspect variables, parameters, model data, and the call sequence at the exact failing line.
⌨️

Q: You're paused in onSave and want to debug inside a helper it calls. Which control?
✓ Step into — it enters the called function so you can walk its lines. Step over would run the helper in one go without going inside it.

Unit 08 · Debugging Data & Events

Binding & Event Debugging

Two of the most common Fiori bugs: data won't display (a binding problem) and a button does nothing (an event handler problem). Both follow a clear checklist.

Binding Bug — a One-Letter Difference
📁 Main.view.xml the model has /Products
<!-- Model data: { "Products": [ { "ProductID": "P101" } ] } --> <Table items="{product>/Products}"> ✅ correct <Table items="{product>/Product}"> ❌ no records appear
Binding Debugging Flow
Six checks, in order — the first "no" is your bug.
Event Bug — Name Mismatch
📁 view + controller the names must match exactly
<Button press=".onSave" /> # view expects onSave onSaveOrder: function(){ ... } ❌ controller has a different name onSave: function(){ ... } ✅ fix: match the handler name
🖱️
"Button does nothing" checklist: is the handler name an exact match? does the controller method exist? is the button disabled? is a transparent overlay blocking clicks? does an exception throw before the action completes? Check Elements (enabled/visible), Sources (breakpoint in handler), Console (exceptions).
Q: A table stays empty; Network shows 200 OK with data and the model is assigned. Next check?
✓ The binding path (and that the response structure matches it). Data arrived and the model exists, so the likely culprit is a wrong path like /Product vs /Products, or a model-name mismatch.
Unit 09 · QUnit — Unit Testing

QUnit Unit Testing

QUnit tests one small piece of logic in isolation — formatters, validators, calculations, helpers. Give an input, assert the output. Structure: Arrange → Act → Assert (AAA).

🏭
Analogy — a phone factory. Before assembling the whole phone, each part is tested alone: battery, camera, speaker, display. QUnit tests each function of your app the same way — small pieces first, so a failure points to exactly one broken part.
The Function Under Test
📁 formatter.js webapp/model/formatter.js
formatStockStatus: function (iStock) { if (iStock <= 0) return "Out of Stock"; if (iStock < 10) return "Low Stock"; return "Available"; }
The QUnit Test (Arrange–Act–Assert)
🧪 formatter test webapp/test/unit/model/formatter.js
QUnit.module("Stock Status Formatter"); QUnit.test("Returns Out of Stock when stock is zero", function (assert) { const sResult = formatter.formatStockStatus(0); // Act assert.strictEqual(sResult, "Out of Stock", "zero → Out of Stock"); // Assert }); QUnit.test("Returns Low Stock below ten", function (assert) { assert.strictEqual(formatter.formatStockStatus(5), "Low Stock"); });
✅ Stock Status Formatter · 3 tests completed · 3 passed · 0 failed
The Four Assertions You'll Use
AssertionChecksExample
strictEqualValue and type matchstrictEqual(x, "Available")
okCondition is truthyok(iTotal > 0)
deepEqualArrays/objects equal by valuedeepEqual(oA, oB)
notStrictEqualValues differnotStrictEqual(s, "")
Boundary Testing — Where Bugs Hide
🎯
Boundary values sit right where a rule changes. For a discount rule (0% below ₹10,000; 5% to ₹49,999; 10% from ₹50,000), test 9999→0, 10000→500, 49999→2499.95, 50000→5000. Most defects happen at these condition limits, not in the middle.
🧪 boundary test discount at the edges
QUnit.test("Discount boundary values", function (assert) { assert.strictEqual(formatter.calculateDiscount(9999), 0); assert.strictEqual(formatter.calculateDiscount(10000), 500); assert.strictEqual(formatter.calculateDiscount(50000), 5000); });
Q: Why test 10000 specifically and not just 25000?
✓ 10000 is a boundary — the exact point the discount rule changes from 0% to 5%. Off-by-one errors (> vs >=) show up right there, while a mid-range value like 25000 would pass even with the bug.
Unit 10 · OPA5 — Integration Testing

OPA5 Integration Testing

OPA5 (One Page Acceptance) drives the real running app like a user — finding controls, pressing, typing, navigating, then asserting what appears. It tests parts working together, structured as Given → When → Then.

🍽️
Analogy — a manager testing the customer experience. The manager acts like a customer: opens the menu, selects food, adds it, confirms, checks the confirmation appears. OPA5 behaves like a real user: open app → find control → perform action → verify result.
Given–When–Then
Given sets the scene, When does the action, Then verifies what the user sees.
A Real Journey
🎭 journey webapp/test/integration/SalesOrderJourney.js
opaTest("Should open the sales-order details page", function (Given, When, Then) { Given.iStartMyApp(); When.onTheSalesOrderList.iPressOnTheFirstOrder(); Then.onTheSalesOrderDetails.theDetailsPageShouldBeVisible(); Then.iTeardownMyApp(); });
Page Objects — Reusable, Readable
🎭 page object test/integration/pages/SalesOrderList.js
Opa5.createPageObjects({ onTheSalesOrderList: { actions: { iPressOnTheFirstOrder: function () { return this.waitFor({ // waits until control is ready id: "salesOrderTable", success: function (oTable) { new Press().executeOn(oTable.getItems()[0]); }, errorMessage: "Sales-order table was not found" }); } } } });
⏳
OPA5 waits for you. UI5 renders asynchronously, so waitFor polls until the control and data are ready before acting — no fragile fixed setTimeout delays. A page object groups a screen's actions and assertions, so if a control ID changes you fix it in one place.
Q: A QUnit test checks the filter-building function. What does the matching OPA5 test add?
✓ It verifies the whole interaction on screen: user types in the search field → search event → filter created → model binding → the table actually shows only matching rows. QUnit checks the piece; OPA5 checks the pieces working together.
Unit 11 · Troubleshooting — Interactive

Errors & Systematic Debugging

Most Fiori problems fall into a handful of patterns. Tap each error to reveal its likely causes and where to look. Then follow the systematic procedure — never change code randomly.

Uncaught ReferenceError: calculateTotal is not defined
▸ tap for cause & fix
Cause: calling a function that doesn't exist or is out of scope. Fix: check the spelling, that it's defined, and that it's reachable from where it's called. Look in the Console.
Cannot read properties of undefined
▸ tap for cause & fix
Cause: reading oCustomer.Name when oCustomer is undefined. Fix: guard it — if (oCustomer) { ... } — and use a breakpoint to see why the object is missing.
Failed to load '.../Main.controller.js'
▸ tap for cause & fix
Cause: wrong namespace, file location, resource-root, dependency path, or a case mismatch in the filename. Fix: verify the path and exact casing; check Network for the failed file.
Blank screen — nothing renders
▸ tap for cause & fix
Cause: JS syntax error, wrong component namespace, missing dependency, invalid manifest.json, or an XML parse error. Fix: Console (JS/XML error) → Network (failed file) → Sources (module path).
Data saves but the UI doesn't update
▸ tap for cause & fix
Cause: model not refreshed, wrong binding mode, response not applied, or UI bound to a different model instance. Fix: ensure the right model updates; getModel().refresh() if appropriate (avoid needless refreshes — the right fix depends on the OData model version).
Frontend or Backend? The Decision Flow
Handler runs? → Request sent? → Response code? Each branch narrows frontend vs backend.
The Systematic Debugging Procedure
1️⃣
Reproduce the issue reliably
2️⃣
Read Console errors
3️⃣
Inspect Network requests
4️⃣
Add a breakpoint
5️⃣
Inspect models & variables
6️⃣
Identify the root cause
7️⃣
Make the smallest correction
8️⃣
Retest the failed scenario
9️⃣
Run regression tests
🧭
Worked example: "Cannot read properties of undefined" on customer select. Breakpoint on the handler shows oEvent exists but selectedItem is undefined — the control actually sends listItem. Fix: read getParameter("listItem"), guard for null, then retest.
Q: The event handler runs and a request is sent, but the response is 500. Frontend or backend?
✓ Backend. A 500 is a server-side exception — the frontend did its job (handler ran, request sent). Investigate the backend service, data, or authorization, not the UI code.
