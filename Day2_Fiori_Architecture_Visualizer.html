Unit 01 · Overview

Fiori Architecture — Animated Overview

Click Send Request to watch a live OData request travel through all three layers — from the browser all the way to SAP HANA and back.

3-Layer Architecture — Live Request Simulation
Three-Layer Summary
🌐
Presentation Layer
SAPUI5 + FLP Shell. Runs in the browser. Displays UI, captures user input, sends OData HTTP requests.
🔗
Integration/API Layer
SAP Gateway. Receives OData HTTP requests, routes to ABAP business logic via MPC/DPC classes.
🗄️
Database Layer
SAP HANA. In-memory columnar database. All business data lives here — read by ABAP via Open SQL/CDS.
📌
Key Principle: The frontend never touches the database. The backend never renders HTML. Each layer communicates only with its adjacent neighbor through a defined interface (OData on the left boundary, Open SQL on the right).

Q: What is the purpose of manifest.json in a SAPUI5 application?
✓ manifest.json is the declarative app descriptor. It defines: routing configuration (URL patterns → view names), OData service endpoints (models section), content density, supported themes, and app metadata. The framework reads it automatically before Component.js runs — no code needed to configure these. Think of it as the app's "settings file."
Unit 03 · Backend

Backend Components

The backend processes every OData request from the frontend. It contains business rules, data validation, and database access — running entirely on the SAP server.

ABAP Backend Transactions
TransactionPurposeUsed When
SEGWGateway Service BuilderCreate OData V2 service · define entities · generate MPC + DPC
/IWFND/MAINT_SERVICEActivate ServicesRegister and activate OData service on the Gateway system
/IWFND/GW_CLIENTGateway Test ClientTest OData GET/POST/PATCH requests (like Postman for Gateway)
PFCGRole MaintenanceAssign catalogs to roles · assign roles to users
SM59RFC DestinationsConfigure Hub → Backend RFC connection
SE80 / SE38ABAP WorkbenchWrite / maintain ABAP code for DPC methods
💡
MPC vs DPC: SEGW auto-generates both. MPC (Model Provider Class) defines the data structure (entity types, properties, associations). DPC (Data Provider Class) implements the behavior — the CRUD methods that call your ABAP business logic. You extend MPC_EXT and DPC_EXT (the "ext" sub-classes) — never modify the base-generated classes directly.
Q: Why should you never modify the base MPC/DPC classes generated by SEGW?
✓ SEGW regenerates the base MPC and DPC classes every time you regenerate the service (e.g., after adding a new entity). Any changes made directly to the base class would be overwritten. SAP provides the _EXT sub-classes (MPC_EXT, DPC_EXT) specifically for custom code — these are never regenerated, so your code is safe there.
Unit 04 · Middleware — Animated

SAP Gateway — Hub vs Embedded

Watch how data flows differently in Hub deployment (separate Gateway server) versus Embedded deployment (Gateway on the same S/4HANA server).

Browser → Gateway Hub (separate server) → Multiple S/4HANA backends via RFC
🏛️
Hub Deployment: Used in large landscapes with multiple SAP backend systems. The Gateway Hub runs on a separate server. The frontend always calls the Hub — never a backend directly. The Hub routes each OData request to the correct backend via RFC (Remote Function Call). The SM59 transaction defines these RFC connection destinations.
Browser → S/4HANA (Gateway + Business Logic on SAME server) — simpler, fewer hops
📦
Embedded Deployment: Gateway runs on the same ABAP server as the business logic. No RFC connection needed — Gateway calls ABAP logic directly in-process. This is the standard for S/4HANA installations. Simpler setup, lower latency, fewer infrastructure components. Frontend still calls one HTTPS endpoint — same as Hub.

Q: What is the difference between Hub and Embedded Gateway deployment?
✓ Hub Deployment: Gateway runs on a separate dedicated server. It connects to multiple backend ABAP systems via RFC (SM59 destinations). Used in large multi-system landscapes. Embedded Deployment: Gateway runs on the same server as the ABAP business logic (standard in S/4HANA). No RFC needed — calls happen in-process. Simpler, lower latency, recommended for new S/4HANA implementations.

Q: What does data-sap-ui-async="true" do and why is it required?
✓ data-sap-ui-async="true" enables asynchronous loading of SAPUI5 libraries and modules. Without it, every library loads synchronously — blocking the browser's main thread and freezing the UI until all resources download. With async=true, resources load in parallel in the background. It is mandatory for all new SAPUI5 apps and is enforced by the UI5 Support Assistant as a critical rule.

Frontend vs Backend — Side by Side

The #1 concept exam questions test. Know exactly what belongs where and why.

AspectFrontend (Browser)Backend (ABAP/BTP)
LanguageJavaScript, XML, JSONABAP, Java/Node.js (CAP)
FrameworkSAPUI5 / OpenUI5RAP / CAP / ABAP MVC
Data accessVia OData HTTP onlyDirect Open SQL / CDS
Business logic❌ None (display only)✅ All rules here

⚠️
Never put business rules in a controller: A user can open DevTools, override your JS validation, and submit anything. The backend must always re-validate. Frontend validation is for UX only — faster feedback, not security.
Q: A max-15-day leave validation — frontend, backend, or both?
✓ Backend is mandatory. Frontend is optional UX enhancement. Anyone can bypass frontend JS with a direct OData POST via Postman or curl. The ABAP DPC/CAP service must validate the 15-day rule and return an OData error response if violated — the frontend then displays that error message to the user.
Unit 07 · Cloud

SAP BTP Architecture

In SAP BTP cloud deployments, the architecture adds Approuter, XSUAA, HTML5 App Repository, and CAP on Cloud Foundry. The browser experience looks identical — the wiring beneath is different.

🚀
Approuter
The entry point. Handles auth redirects to IAS, injects JWT tokens, routes to HTML5 repo / CAP service.
🔐
XSUAA
OAuth 2.0 server on BTP. Issues JWT tokens. Defines scopes. Replaces PFCG for cloud authorization.
🌐
HTML5 App Repo
Managed static file host for SAPUI5 bundles (JS, XML, CSS). No web server to manage.
☕
CAP Service
Cloud Application Programming model — Node.js or Java OData service replacing Gateway on BTP.
📡
Cloud Connector
Secure tunnel to on-premise systems. No inbound firewall ports needed — outbound only.
🗄️
HANA Cloud
Fully managed cloud HANA instance. CAP services connect to it for persistence.
Q: What replaces SAP Gateway in a BTP CAP application?
✓ In a BTP CAP application, CAP (Cloud Application Programming Model) replaces SAP Gateway. CAP automatically exposes your CDS entity definitions as OData V4 services — no SEGW transaction, no MPC/DPC classes. You define entities in .cds files, annotate them with @odata, and CAP generates the full OData service. Authorization is handled by XSUAA (not PFCG).

Unit 09 · Review

Quiz & Self-Assessment

Six questions covering everything in Day 2. Answer all before moving to Day 3.

Q1: Name the three layers of Fiori architecture and one component per layer.
✓ Presentation: SAPUI5 / FLP Shell (browser) · Integration/API: SAP Gateway / CAP (OData middleware) · Database: SAP HANA (in-memory columnar store)
Q2: What does ICF do and where is it configured?
✓ ICF (Internet Communication Framework) is the HTTP service registration system in SAP NetWeaver. Every OData service is registered as a node under /sap/opu/odata/. ICF routes incoming HTTP requests (via ICM) to the correct ABAP handler class. Configured via transaction SICF; services are activated there and in /IWFND/MAINT_SERVICE.
Q3: What does sap.ui.define() do?
✓ sap.ui.define() declares a UI5 AMD module. It takes: (1) an array of dependency paths, (2) a factory function receiving the loaded dependencies. The UI5 loader resolves all dependencies asynchronously before calling the factory. This prevents missing-dependency errors and enables parallel resource loading.
Q4: Hub vs Embedded — name one advantage of each.
✓ Hub advantage: Can aggregate OData services from multiple backend systems behind one endpoint — ideal for multi-system landscapes. Embedded advantage: Simpler setup (no separate server, no RFC config), lower latency (in-process calls), standard for modern S/4HANA.
Q5: What are the four controller lifecycle hooks in order?
✓ onInit() — once when view created (no DOM yet) · onBeforeRendering() — before each render · onAfterRendering() — after each render (DOM available) · onExit() — when view destroyed (cleanup)
Q6: In BTP, what replaces PFCG roles and SAP Gateway?
✓ XSUAA (Extended Services User Account and Authentication) replaces PFCG roles for authorization on BTP — it issues JWT tokens with scopes. CAP (Cloud Application Programming Model) replaces SAP Gateway — CAP auto-exposes CDS entity definitions as OData V4 services without SEGW or MPC/DPC classes.
