Unit 01 · Foundation

Why Roles & Catalogs?

You've built apps. Now the question is: who can see them, and where do they appear? Day 10 is about the Fiori authorization & content model — the machinery that decides which tiles land on which user's launchpad.

🔑
In simple words — an office building: the apps are rooms. A catalog is the master list of all rooms that exist. A group is a floor directory showing a handy set of rooms. A role is the keycard that actually unlocks certain rooms. You need the room to exist and a keycard that opens it.

Flow Diagram — The Access Chain
Every tile reaches a user only when all five links are present.
The Three Building Blocks (Today's Subtopics)
📚
Catalog
The full menu of apps a user is allowed to find (via App Finder).
🗂️
Group
A tidy set of tiles shown together on the home page for quick access.
🪪
PFCG Role
The backend authorization that ties catalogs/groups to a user.
🧭
The key relationship: a role references one or more catalogs and groups. Assign that role to a user, and they get every tile inside — findable (catalog) and visible (group). No role → no tiles, however many catalogs exist.
Q: A catalog with your app exists, but a user sees no tile. Most likely reason?
✓ The user has no role connecting them to that catalog. A catalog just makes apps available; a role assigned to the user is what actually grants access.
Unit 02 · Building Blocks

Tiles & Semantic Objects

Before catalogs, understand what goes into them: tiles (what users tap) and the semantic object + action that tells Fiori what to launch. This is the "target mapping" from Day 3, revisited.

🔲 Tile
The clickable card on the launchpad. Has a title, icon, maybe a live number. Tapping it triggers a navigation intent.
🎯 Target Mapping
The rule that says "this intent → launch that app." Without it, a tile has nowhere to go.
Flow Diagram — What Happens When a Tile Is Tapped
A tile never points at a URL — it fires an intent that the target mapping resolves.
Semantic Object + Action = Intent

Fiori navigation is intent-based. An intent is a business meaning, not a URL — which is why the same tile works across apps and devices.

# Intent = SemanticObject - Action Product-display # "show a product" SalesOrder-create # "create a sales order" Employee-manage # "manage an employee" # A tile fires an intent → target mapping resolves it → app opens
📮
Analogy — a postal address vs a person's name: an intent like Product-display is like asking for "the Sales Manager" (a role/meaning), not "desk 4B, floor 2" (a fixed URL). If the person moves desks, "Sales Manager" still finds them. Intents survive app changes the way a title survives a desk move.
How the Pieces Nest
PieceWhat it isLives in
TileThe launch button + its intentA catalog
Target MappingIntent → app resolution ruleA catalog
Semantic ObjectThe business "thing" (Product, Order)Defined centrally, used in intents
Q: Why does Fiori use intents like Product-display instead of direct app URLs?
✓ Because intents express business meaning, not a fixed location. Target mappings resolve an intent to whatever app currently handles it — so tiles, links, and cross-app navigation keep working even if the underlying app changes.

Q: A tile is in a user's group but not in any catalog they have. What happens?
✓ It won't work properly — the group controls visibility, but the catalog controls access. Without the catalog authorization, the user can't actually launch it. Always back a group tile with a catalog.
Unit 04 · Authorization

What Is a PFCG Role?

PFCG is the ABAP transaction for the Role Maintenance tool. A role bundles authorizations and — for Fiori — references the catalogs and groups a user should get. It's the bridge between "content exists" and "this person can use it."

🪪
In simple words — a keyring: a PFCG role is a labelled keyring ("Sales Rep"). On it hang the keys (catalogs/groups) that a Sales Rep needs. Hand the same keyring to every sales rep and they all get exactly the right doors — manage one keyring, not hundreds of individual keys.
Flow Diagram — What a Fiori Role Bundles
The role is the hub: content + permissions on one side, users on the other.
What a Fiori PFCG Role Contains
📚
Catalog reference
Which app catalogs this role grants (the "can access" list).
🗂️
Group reference
Which tile groups appear on the home page.
🔐
Authorizations
Backend permissions (OData services, objects) auto-pulled in.
👥
Assigned users
The people who receive everything above.
Two Flavours of Role
Role typeWhat it isContains
Single roleOne focused set of accessCatalogs, groups, authorizations
Composite roleA bundle of single rolesSeveral single roles together
🧩
Composite = keyring of keyrings. A "Sales Manager" composite role might bundle "Sales Rep" + "Reports Viewer" + "Team Lead" single roles — so one assignment gives all three sets. Great for job profiles.
Q: What transaction do you use to maintain roles in the ABAP backend?
✓ PFCG — the Role Maintenance transaction. It's where you create the role, add Fiori catalogs/groups, generate the authorization profile, and assign users.
Unit 05 · PFCG — Animated

Building a Role in PFCG

Here's the real sequence a security admin follows in transaction PFCG to create a Fiori role. Press play to walk through it.

① Open PFCG & create the role
Enter a role name (e.g. Z_SALES_REP) and click Create. Add a description.
② Menu tab → add Catalog
Insert the Fiori catalog (SAP Fiori Tile Catalog) — the apps this role may access.
③ Menu tab → add Group
Insert the Fiori group (SAP Fiori Tile Group) — the tiles shown on the home page.
④ Authorizations tab → generate
Click the profile button; PFCG pulls in the needed authorizations (OData services, objects) and you generate the profile.
⑤ User tab → assign users
Add the business users, then User Comparison to push the profile to them.
⑥ Done — user logs in
On next login the user sees the group's tiles and can find catalog apps in the App Finder.
🔄
Don't forget User Comparison. After assigning users you must run User Comparison (the "Compare" button) so the generated authorization profile is actually written to each user's master record. Skipping it is a classic "role assigned but nothing works" cause.
Q: In PFCG, which tab do you use to add the Fiori catalog and group?
✓ The Menu tab — you insert the "SAP Fiori Tile Catalog" and "SAP Fiori Tile Group" there. The Authorizations tab then generates the matching backend permissions, and the User tab assigns people.

Q: What's the correct order: role, catalog, tile, user?
✓ App → Tile → Catalog (+ optional Group) → Role → User. You build the content first, wrap it in a catalog, reference the catalog from a role, then assign the role to the user.

Q: In the hierarchy, what sits directly inside a catalog?
✓ Tiles and their target mappings. The catalog holds the launchable items (tiles) plus the rules that resolve each tile's intent to an app. The catalog itself sits inside a role.
Unit 08 · Troubleshooting — Interactive

Common Pitfalls & Fixes

Almost every "my tile won't show / won't open" problem is one of these. Tap each to see the real cause and fix.

Flow Diagram — Debug in This Order
Walk the chain in order; stop at the first step that isn't satisfied.
Tile assigned but user sees nothing
▸ tap for cause & fix
Cause: the role was assigned but User Comparison wasn't run, so the profile never reached the user. Fix: in PFCG → User tab, click Compare, then have the user re-login.
Tile shows but clicking it does nothing / error
▸ tap for cause & fix
Cause: missing or wrong target mapping — the intent can't resolve to an app. Fix: in the catalog, check the target mapping's semantic object, action, and the app's technical details.
App opens but "not authorized" for data
▸ tap for cause & fix
Cause: the OData service authorization wasn't included/generated. Fix: in PFCG → Authorizations, re-generate the profile so the service's auth objects are pulled in.
Tile in group but greyed out / missing
▸ tap for cause & fix
Cause: the tile is in a group but not in a catalog the user has. Fix: ensure the same tile is in an authorized catalog — group = visibility, catalog = access.
🩺
Debug order: check the chain in sequence — target mapping → catalog membership → role reference → user assignment → User Comparison → re-login. The first broken link is your culprit.
Q: Role assigned, catalog correct, but the tile still doesn't appear. What's the classic missing step?
✓ User Comparison (and re-login). Assigning the role isn't enough — the generated authorization profile must be pushed to the user's master record via Compare in PFCG.
