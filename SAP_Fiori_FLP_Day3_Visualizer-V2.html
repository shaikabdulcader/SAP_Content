Part 01 · Foundation

What is the Fiori Launchpad?

The single most common beginner mistake: thinking FLP is a Fiori application. It is not. FLP is the container that hosts all Fiori applications.

❌
Myth: "FLP = a Fiori Application."  Reality: FLP is a shell / container. It launches apps but is not itself a business app.

Q: Why is FLP called a "shell" instead of an application?
✓ Because FLP is the hosting container that provides cross-app services — home page, search, notifications, intent navigation, personalization, theming — and launches business apps inside it. It contains no business logic of its own, exactly like an operating system shell hosts programs.

2 · Dynamic Tile ⭐ (Interview Favourite)
📊
A manager with 47 pending approvals sees the number on the tile itself — no need to open the app. Tomorrow it may read 52, then 18. The number updates automatically.
Where does that number come from?
# The tile periodically calls an OData service: GET /sap/opu/odata/sap/ZWFAPPROVAL_SRV/ApprovalSet/$count?$filter=Status eq 'Pending' # Breakdown GET → read data ApprovalSet → entity holding approvals $filter → only pending ones $count → return ONLY the number # Response is a PLAIN number — not JSON: 47 ✓ (NOT { "count": 47 })
⚠️
A Dynamic Tile expects the raw scalar 47, not an object {"count":47}. That's exactly what $count returns — which is why Dynamic Tiles use it.

Q: Why does a Dynamic Tile use the $count OData query option?
✓ Because a Dynamic Tile only needs a single number to display, and $count returns exactly that — a plain scalar value (e.g. 47) rather than a full JSON payload. It's lightweight, fast to poll periodically, and matches the exact format the tile's data source expects.
Part 03 · Navigation

Intent-Based Navigation

One of the most important concepts in SAP Fiori. When you click a tile, FLP does not open a hardcoded URL. It creates an Intent and resolves it to the right app.

❌
Common misconception: "Click Tile → Open URL."  SAP never opens a hardcoded URL directly. Instead it generates an Intent.
What is an Intent? — Build one (click chips)
#LeaveRequest-create
Semantic Object LeaveRequest Employee SalesOrder Product
Action create display manage approve
Reading: Open → LeaveRequest → in create mode
General Format
Semantic Object · Noun
#SemanticObject
The business thing (Employee, SalesOrder…)
-
Action · Verb
Action
What you do to it (display, manage…)
Full Navigation Flow (click each step)
🖱️ User clicks 📝 Leave Request tile
↓
🧭 FLP generates intent #LeaveRequest-create
↓
📡 Navigation Service catches the intent
↓
🔎 Reads Semantic Object + Action
↓
🗺️ Searches Target Mapping
↓
🔐 Checks authorization
↓
⚛️ Loads SAPUI5 app + passes parameters
↓
✅ Application opens
Step 1 — Click: The user taps the Leave Request tile. Nothing is hardcoded yet — no URL is opened.
Step 2 — Intent generated: FLP builds the intent #LeaveRequest-create from the tile's configuration. Think of it as a business command, not an address.
Step 3 — Navigation Service: The FLP Navigation Service intercepts the intent and takes ownership of resolving it.
Step 4 — Parse: It splits the intent into Semantic Object = LeaveRequest and Action = create.
Step 5 — Target Mapping lookup: It searches the Target Mappings for a matching Object + Action pair to find which application should handle it (covered in Part 4).
Step 6 — Authorization: It confirms the user is authorized (via their Role/Catalog) to open the resolved application.
Step 7 — Load: The matched SAPUI5 application is loaded and any parameters (e.g. mode = create) are passed to it.
Step 8 — Open: The application renders in create mode. The user never saw a URL — only a business intent was expressed. ✓
Real-Life Analogy — Calling Uber
🙅 You DON'T say
"Drive Car #42"
✕
Open /sap/bc/ui5_ui5/sap/zleaveapp
Hardcoded & brittle — avoid.
🙆 You DO say
"Take me to the Airport"
✓
#LeaveRequest-create
Express intent — FLP decides the app.
🚕
Uber decides which driver, which route, which car. Likewise, FLP decides which application handles your intent. You express what you want, not how to reach it.
Q: What is Intent-Based Navigation?
✓ A navigation mechanism where FLP resolves an intent (#SemanticObject-Action) to the correct application using Target Mapping — instead of opening a hardcoded URL. This decouples navigation from technical app URLs, so apps can be swapped or upgraded without breaking every link.
Part 04 · Routing

Target Mapping

If an Intent is "where I want to go," Target Mapping is the route that gets you there. Think of it as Google Maps for FLP — it maps an intent to a concrete application.

What Happens Without Target Mapping?
🔲 Tile
↓
🧭 Intent #LeaveRequest-create
↓
❌ No App Found — navigation fails
💥
Without a Target Mapping, the intent resolves to nothing. The tile exists, the intent is generated — but FLP has no route, so navigation fails. Target Mapping is the mandatory link between a tile and a real app.
Why Priority? (see it resolve live in Part — Priority Deep Dive)
# Two apps can both create leave requests: App A Priority 100 App B Priority 50 # FLP always chooses the HIGHER priority: → App A wins (100 > 50) # If both are equal → behaviour is not deterministic (see Unit 9)
Q: What is Target Mapping?
✓ A routing configuration that maps an intent (Semantic Object + Action) to a specific application — defining the app type, URL, and priority. Without it, an intent cannot be resolved and navigation fails. It's the "route" half of Intent-Based Navigation.
Part 05 · Access

Catalog · Role · Group

This is where beginners get confused. There are actually two separate chains: one decides if you can use an app, the other decides if you can see it on your home page.

The Two Chains (they answer different questions)

↓

Answers: CAN I USE IT?
👁️ Chain 2 — Visibility
Tile
↓

↓

The Netflix Analogy
🎬
You bought Netflix Premium → you're authorized to watch any movie. But if you haven't added a film to "My List," it won't show on your home page. You can still search for it and watch it. Authorization ≠ Visibility.
Scenario — Authorized but Invisible
✅ Has HR Role
Employee → HR Role → HR Catalog
Authorized: YES
🚫 Tile in no Group
Home Page → (no tile shown)
Visible on home: NO
🔎
The user can still find and launch it via the App Finder, because permission comes from the Catalog. The Group only controls whether it appears on the home page.
Easy Way to Remember
📚
Catalog → Permission
"Can I use it?" Controls authorization — the set of apps a role is allowed to use.
🗂️
Group → Visibility
"Can I see it on my home page?" Controls layout — how tiles are arranged and shown.
Q: What is the difference between a Catalog and a Group?
✓ A Catalog controls authorization — which apps a user (via their role) is allowed to use. A Group controls visibility — which tiles appear on the home page and how they're arranged. A user can be authorized for an app (Catalog) yet not see its tile on the home page (no Group) — but can still launch it from the App Finder.

Part 07 · End-to-End

Complete Tile Configuration

Your manager says: "Create a new Leave Request tile." Here's the real project sequence — click each step to open it.

The 6-Step Configuration Sequence
1
🗺️ Target Mapping
Create the mapping: LeaveRequest → create → SAPUI5 Application. This is the route that connects the future intent to the actual app.
2
🔲 Tile
Create the tile — Title: Leave Request, Icon: 📅, Intent: LeaveRequest-create. This is what the user will click.
3
📚 Catalog
Create the HR Self Service Catalog and add the Leave Request tile to it. Now the tile is available to be assigned through roles (authorization).
4
👔 Role (PFCG)
Assign the Catalog to the HR Employee Role. Now anyone who receives this role is authorized to use the application.
5
🧑 User
Assign the HR Employee Role to a specific user, e.g. Madhesh.P. Now that user has access.
6
🗂️ Group
Create the HR Self Service group and add the Leave Request tile. Now it finally appears on the user's FLP home page.
Complete Lifecycle Picture
🧑‍💻 Developer creates the app
↓
🗺️ Target Mapping
↓
🔲 Tile
↓
📚 Catalog
↓
👔 Role (PFCG)
↓
🧑 User
↓
🗂️ Group
↓
🏠 FLP Home Page ✓
🔗
Why this order matters: Target Mapping first (so the intent resolves), then the Tile (something to click), then Catalog + Role + User (authorization chain), and finally the Group (visibility). Skip the Group and the app still works — it just won't show on the home page.
Q: Explain the complete flow from clicking a tile to opening a SAPUI5 application.
✓ The user clicks a tile → FLP generates an intent #SemanticObject-Action → the Navigation Service parses it → it looks up the matching Target Mapping → checks the user's authorization (Role → Catalog) → loads the mapped SAPUI5 application, passing parameters → the app opens. The tile only appears on the home page if it's also placed in a Group.

Target Mapping Priority

A common real-world question: when several Target Mappings match the same intent, which app wins? Drag the priorities below and watch FLP resolve the winner live.

Interactive Priority Resolver
All three apps respond to #LeaveRequest-create. Adjust priorities — higher number wins.
🟩 Custom Leave App 100 —
🟦 SAP Standard Leave App 10 —
🟧 Legacy Fallback App 50 —
🧪
Try this: set two apps to the same highest value and watch the resolver flag a tie — that's exactly the non-deterministic case SAP warns against.
How Priority Works
AppPriorityResult
App A100✅ Selected
App B90Ignored
App C50Ignored
🔢
Priority is a plain integer — higher wins. Values are relative: 5 beats 4, and 1000 beats 999. SAP defines no fixed scale (100=High etc.) — only the ordering matters.
The Tie Case — Equal Priority
⚠️
If two mappings share the same Semantic Object, Action and Priority, FLP cannot decide by priority. Depending on version/config it may: (1) show an App Selection (Resolver) dialog, (2) pick one via internal ordering (not reliable), or (3) warn admins about duplicate mappings.
Recommended Project Convention
ScenarioRecommended Priority
Company custom app (Z / Y app)100
SAP standard app10 or 50
Deprecated / fallback appLower than the preferred app
Where You Set It
# In Launchpad Designer (older) or Content Manager / Spaces & Pages (newer) Target Mapping Semantic Object : LeaveRequest Action : create Application Type: SAPUI5 URL : /sap/bc/ui5_ui5/sap/zleaveapp Priority : 100 ← just enter an integer & save
✅
Best practice: Avoid multiple Target Mappings for the same Semantic Object + Action unless you intentionally want several apps for that intent — and if you do, give them distinct priorities so the preferred (usually custom Z) app always wins.
Q: What happens if two Target Mappings have the same Semantic Object, Action, and Priority?
✓ FLP finds multiple matching applications but cannot distinguish them by priority. Depending on system configuration it may present an app-selection dialog or resolve via internal logic — which should not be relied upon. SAP best practice is to assign different priorities or avoid duplicate mappings for the same intent.
