Unit 01 · Foundation

What Is SAPUI5?

SAPUI5 is SAP's JavaScript UI framework for building responsive enterprise web apps. It's the technology that actually renders every Fiori app you see in the Launchpad — the buttons, tables, and forms are all UI5 controls.

Why SAPUI5 Matters
FeatureWhat It Gives You
Rich control library100s of ready-made UI pieces — buttons, tables, charts, dialogs
ResponsiveOne code base adapts to phone, tablet, and desktop automatically
MVC patternClean separation of screen (View), logic (Controller), and data (Model)
Data bindingConnect UI to data once — it updates itself when data changes
Fiori themingBuilt-in themes (e.g. Horizon) give the consistent Fiori look
📌
SAPUI5 vs OpenUI5: they're the same framework. OpenUI5 is the free open-source core; SAPUI5 is SAP's licensed version with extra enterprise controls (like the smart/analytical controls) and official support.
Q: In one sentence, what does SAPUI5 do?
✓ SAPUI5 is SAP's JavaScript framework that builds and renders the user interface of Fiori apps — providing ready-made responsive controls, the MVC structure, and data binding to OData services.
Unit 02 · Controls — Interactive

UI5 Controls

A control is a reusable UI building block — a button, an input, a list. Each one is a JavaScript object with properties (how it looks) and events (what it does). Click a control below to see it live and read its XML.

Two Ways to Create a Control
✅ XML View (recommended)
<Button text="Press Me" press=".onPress" />
JavaScript (imperative)
new Button({ text: "Press Me", press: onPress });
💡
Prefer XML views. They're declarative (easy to read), tooling-friendly, and keep the UI separate from logic. You'll write the vast majority of your screens in XML and put behaviour in the controller.
Q: What's the difference between a control's property and its event?
✓ A property describes how the control looks or what state it's in (e.g. text, enabled, value). An event is something the user triggers that runs your code (e.g. press, change). Properties are nouns; events are verbs.
Unit 03 · Architecture — Animated

MVC — Model, View, Controller

SAPUI5 organises every app into three parts so responsibilities stay separate. Press play to see how a user action flows through all three.

Three parts, one job each. Press play to follow a button click all the way through and back.
PartFile TypeResponsibility
View.view.xmlWhat the user sees — layout & controls. No logic.
Controller.controller.jsWhat happens — event handlers, calls to the model.
ModelJSON / ODataThe data — held in a model, bound to the view.
🧭
Why separate them? A designer can change the View (XML) without breaking logic; a developer can change the Controller without touching the layout; and the Model can switch from test JSON to live OData with no view changes. Same idea as the Client-Server split you saw on Day 4.
Q: Where does the code that runs when a user presses a button live?
✓ In the Controller (.controller.js). The View's button declares press=".onPress", and the matching onPress function sits in the controller. The View stays purely visual.

Data Binding

Binding connects a control to a piece of data once, and UI5 keeps them in sync automatically. Change the data below and watch the view update instantly — no manual DOM code.

The {/name} in the view is a binding path — it points at the name field in the model. Type above and the view refreshes on its own.

The Common Model Types
ModelUse ForSource

Binding Syntaxes You'll See
SyntaxNameMeaning
text="{/name}"Property bindingBind one control property to one field
items="{/products}"Aggregation bindingRepeat a template for each item in a list
{i18n>appTitle}Named model bindingRead from a specific named model
🔗
Two-way binding means it works both directions: if the user edits an Input bound to {/name}, the model updates too — exactly like the demo above. This is the magic that removes most manual UI code.
Q: You need to show a table of leave requests from your SAP backend. Which model?
✓ ODataModel — it connects directly to the OData service you registered on Day 4. You'd bind the table's items to the entity set (e.g. items="{/LeaveRequestSet}") and each row fills in automatically.
Unit 05 · Development Tools

Web IDE vs Business Application Studio

These are the environments where you write UI5 apps. SAP Web IDE is the older tool; Business Application Studio (BAS) is its modern successor and the one you'll use today.

🧓
SAP Web IDE
The predecessor
Legacy
  • Browser-based IDE on the older Neo environment
  • Was the standard tool for years
  • No longer sold to new customers
  • Planned sunset on Dec 31, 2028
  • You may still meet it on existing projects
🚀
Business Application Studio
The current standard
Use This
  • Cloud IDE on SAP BTP, built on VS Code
  • Default for new customers since 2020
  • "Dev spaces" pre-loaded for Fiori/UI5
  • Project wizards, guided templates, deploy tools
  • Integrated preview, debugging & Git
🧭
The short version: if the syllabus says "Web IDE / BAS", think of them as old vs new versions of the same idea — a browser-based place to build UI5 apps. New work happens in BAS. SAP is also expanding UI5 tooling into VS Code extensions.
Typical First-App Flow in BAS
①
Open a Dev Space — pick the "SAP Fiori" kind
②
New Project from Template — choose SAPUI5 freestyle app
③
Pick a data source — connect an OData service (or none)
④
Run / Preview — test in the browser instantly
⑤
Deploy — push to BTP or the ABAP repository
Q: A new project is starting today. Which tool should the team use — Web IDE or BAS?
✓ Business Application Studio (BAS). Web IDE is legacy and heading for retirement in 2028; BAS is the current, supported environment for all new UI5/Fiori development (with VS Code extensions as an emerging option).

The two files people always ask about: manifest.json is the app's "descriptor" — its ID, data sources, routing, and models all declared in one place. Component.js is the entry component that boots the app. Together they replace a lot of manual wiring.
Q: Which file declares the app's data sources, models, and routing config?
✓ manifest.json — the application descriptor. It's the single source of truth for the app ID, OData data sources, model definitions, and routing/target configuration.
Unit 07 · Startup — Animated

Bootstrap & App Lifecycle

How does a UI5 app actually start? It begins with a bootstrap script in index.html that loads the framework, then the component and first view. Press play to watch it boot.

The bootstrap tag in index.html
<script id="sap-ui-bootstrap" src="resources/sap-ui-core.js" data-sap-ui-theme="sap_horizon" data-sap-ui-libs="sap.m" data-sap-ui-on-init="module:myApp/index"> </script>
📜
index.html loads — the browser reads the bootstrap script tag
⚙️
Core loads — sap-ui-core.js + the declared libraries (e.g. sap.m)
🎨
Theme applied — the Horizon theme styles everything
🧩
Component.js runs — reads manifest.json, sets up models & routing
🖼️
Root view renders — the first View + Controller appear on screen
✅
App is ready — user can interact; data binding is live
Controller Lifecycle Hooks

Inside a controller you can hook into 4 moments in a view's life:

HookFires When…Common Use
onInitview is created (once)set up models, read route args
onBeforeRenderingjust before it's drawnlast-minute prep
onAfterRenderingright after it's in the DOMDOM tweaks, 3rd-party libs
onExitview is destroyedclean up timers/listeners
Q: Where would you load initial data when a view first opens?
✓ In onInit — it runs once when the controller's view is created, making it the natural place to set up models and load the data the screen needs.

Unit 09 · Review

Quick-Fire Revision

Five questions for tomorrow. Try to answer out loud before revealing.

Q1: What is SAPUI5, in one line?
✓ SAP's JavaScript framework that builds and renders the UI of Fiori apps — responsive controls, MVC, and data binding.
Q2: Name the three parts of MVC and what each does.
✓ View = the screen (XML). Controller = the logic/event handlers (JS). Model = the data (JSON/OData), bound to the view.
Q3: What does data binding give you?
✓ An automatic link between a control and a data field — change the data and the UI updates itself (and, with two-way binding, vice versa).
Q4: Web IDE or BAS for new work — and why?
✓ BAS. Web IDE is legacy (sunset 2028); BAS is the current cloud IDE on BTP, built on VS Code, default for new customers.
Q5: Which file is the app descriptor, and what's in it?
✓ manifest.json — holds the app ID, data sources, model definitions, and routing config in one place.

One-line summary: SAPUI5 builds Fiori screens from ready-made controls, organised with MVC, kept in sync by data binding, written today in Business Application Studio — and every app shares the same file structure anchored by manifest.json.
