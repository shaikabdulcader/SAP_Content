Unit 01 · Foundation

What Is OData?

OData (Open Data Protocol) is an HTTP-based standard for exposing and consuming business data. In SAPUI5, an OData model connects your screen to backends like S/4HANA, SAP Gateway, CAP, BTP services, or any database exposed through OData.

🍽️
In simple words — a restaurant: your app never walks into the kitchen itself. It tells the waiter (the OData service) what it wants; the waiter carries the order to the kitchen (the backend) and brings the food (data) back. Your app just talks to the waiter.

The Overall Architecture

Data flows down through clear layers and comes back up. Your event handlers sit in the controller; the model does the HTTP work.

🔑
The one idea to hold onto: the app does not touch the database directly. Every read or change is an HTTP request sent through the OData service. Learn that path and everything else clicks into place.
Q: In the restaurant analogy, what plays the role of the "waiter"?
✓ The OData service. The app (customer) sends an order (HTTP request) to the waiter (OData service), who fetches from the kitchen (backend/database) and returns the food (data). The app never enters the kitchen itself.
Unit 02 · Setup

The Service & Model Setup

Before any CRUD, two things exist: an OData service exposing data (like a Products entity set), and an OData model in your app configured to talk to it.

Anatomy of a Service URL

Say an inventory service exposes a Products entity set, where each product has ProductID, ProductName, Category, Price, Stock.

You want…The URL
All productsGET /odata/v2/InventoryService/Products
One product by keyGET .../Products('P1001')
The "menu" (metadata)GET .../$metadata
📋
What is $metadata? It's the machine-readable menu: it lists every entity set, each property and its data type, the primary keys, relationships, and available actions. SAPUI5 reads it so it knows how to build requests correctly.
Configuring the Model in manifest.json

You declare the service as a data source, then create a model that uses it. An empty model name "" makes it the default model.

📄 manifest.json
{ "sap.app": { "dataSources": { "mainService": { "uri": "/odata/v2/InventoryService/", "type": "OData", "settings": { "odataVersion": "2.0" } } } }, "sap.ui5": { "models": { "": { // "" = default model "dataSource": "mainService", "settings": { "defaultBindingMode": "TwoWay", "useBatch": true } } } } }
🎯
Because it's the default model, you grab it in any controller with one line — no name needed:
const oModel = this.getView().getModel();
Q: What does the empty model name "" in manifest.json mean?
✓ It marks this as the default model, so you can access it with this.getView().getModel() (no name) and bind controls with plain paths like {ProductName} — no model prefix required.

Why MERGE for update? The V2 model's default update uses MERGE, which changes only the properties you send instead of overwriting the whole record. Send just {Price: 949} and the other fields stay untouched.
Q: You send {Stock: 20} via update(). What happens to ProductName?
✓ It stays unchanged. Because V2 update uses MERGE, only the properties you include are modified — the fields you leave out keep their existing values.
Unit 04 · Create — Animated + Code

CREATE: Inserting a New Record

Create inserts a new record — like registering an employee, adding a product, or submitting a leave request. The form values become the request payload. Press play to trace it.

📝

🧱 Main.view.xml — declare the form & button
<VBox class="sapUiMediumMargin"> <Input id="productIdInput" placeholder="Product ID"/> <Input id="productNameInput" placeholder="Product name"/> <Input id="priceInput" type="Number" placeholder="Price"/> <Button text="Create Product" type="Emphasized" press=".onCreateProduct"/> </VBox>
⚙️ Main.controller.js — the handler
onCreateProduct: function () { const oModel = this.getView().getModel(); const oProduct = { ProductID: this.byId("productIdInput").getValue().trim(), ProductName: this.byId("productNameInput").getValue().trim(), Price: Number(this.byId("priceInput").getValue()) }; if (!oProduct.ProductID || !oProduct.ProductName) { MessageBox.warning("Product ID and name are required."); return; } oModel.create("/Products", oProduct, { success: function () { MessageToast.show("Product created."); }, error: function (oError) { MessageBox.error("Create failed."); } }); }
📡 Resulting HTTP request
POST /odata/v2/InventoryService/Products Content-Type: application/json { "ProductID": "P1001", "ProductName": "Wireless Mouse", "Price": 899, "Stock": 50 }
Q: Where do the values in the POST body come from?
✓ From the form inputs. The handler reads each field with this.byId("...").getValue(), builds a product object, and passes it as the second argument to oModel.create() — which becomes the request payload.
Unit 05 · Read — Two Ways + Code

READ: Getting Data onto the Screen

Read retrieves data — displaying a product list, viewing orders, searching. There are two ways: let binding do it (usual), or call read() explicitly when you need control.

📚
Analogy: Read is like asking a librarian for a book. You give search criteria; they return the matching book — or a whole list.
Method 1 — Data Binding (preferred for lists/tables)

Bind a table to the entity set and the model does the GET, fills rows, and keeps them updated. No request code.

🧱 Main.view.xml
<Table id="productTable" items="{/Products}" mode="SingleSelectMaster" selectionChange=".onProductSelect"> <columns> ... </columns> <items> <ColumnListItem> <cells> <Text text="{ProductID}"/> <Text text="{ProductName}"/> <ObjectNumber number="{Price}" unit="INR"/> </cells> </ColumnListItem> </items> </Table>
🔗
The binding items="{/Products}" makes the model send GET /Products automatically and drop one row in per record. This is why Read usually needs the least code.
Method 2 — Explicit read()

Use when you need custom processing, a count/summary, or to copy results into a JSON model.

⚙️ Main.controller.js
onLoadProducts: function () { const oModel = this.getView().getModel(); this.getView().setBusy(true); oModel.read("/Products", { success: function (oData) { MessageToast.show(oData.results.length + " products loaded."); this.getView().setBusy(false); }.bind(this), error: function () { this.getView().setBusy(false); }.bind(this) }); }
Reading One Record — use createKey()
const sPath = "/" + oModel.createKey("Products", { ProductID: "P1001" }); oModel.read(sPath, { success: function (oProduct) { console.log(oProduct); } });
🔐
Why createKey()? It builds the key path safely — /Products('P1001') — with correct quoting and encoding, so you don't hand-glue strings and risk typos or special-character bugs.
Searching = Filtering the Binding
⚙️ onSearchProduct handler
onSearchProduct: function (oEvent) { const sQuery = oEvent.getParameter("newValue"); const oBinding = this.byId("productTable").getBinding("items"); const oFilter = new Filter("ProductName", FilterOperator.Contains, sQuery); oBinding.filter(sQuery ? [oFilter] : []); }
📡 May generate
GET /Products?$filter=contains(ProductName,'Mouse')
Q: To show a product table on screen, which Read method needs the least code?
✓ Data binding — items="{/Products}". The model issues the GET and fills the rows for you. Use explicit read() only when you need custom handling of the results.
Unit 06 · Update & Delete — Code

UPDATE & DELETE

Both operate on one existing record, identified by its key. Update changes selected fields; Delete removes it (and should always be confirmed first).

✏️ UPDATE
Like correcting the phone number on an existing employee record — you change selected info, not create anew.
🗑️ DELETE
Like removing an outdated file from a cabinet. It may be irreversible — always confirm with the user.
Step 1 — Selection Fills the Form (an event!)

When the user picks a row, grab the record from its binding context and populate the form.

⚙️ onProductSelect
onProductSelect: function (oEvent) { const oItem = oEvent.getParameter("listItem"); const oProduct = oItem.getBindingContext().getObject(); this.byId("productNameInput").setValue(oProduct.ProductName); this.byId("priceInput").setValue(oProduct.Price); }
Step 2 — Update the Changed Fields
⚙️ onUpdateProduct
onUpdateProduct: function () { const oModel = this.getView().getModel(); const oExisting = this.byId("productTable").getSelectedItem() .getBindingContext().getObject(); const sPath = "/" + oModel.createKey("Products", { ProductID: oExisting.ProductID }); const oChanged = { Price: Number(this.byId("priceInput").getValue()) }; oModel.update(sPath, oChanged, { merge: true, success: function () { MessageToast.show("Updated."); } }); }
📡 HTTP
MERGE /odata/v2/InventoryService/Products('P1001') { "Price": 949, "Stock": 75 } // only these change
Delete — Always Confirm First
⚙️ onDeleteProduct
onDeleteProduct: function () { const oModel = this.getView().getModel(); const oProduct = this.byId("productTable").getSelectedItem() .getBindingContext().getObject(); const sPath = "/" + oModel.createKey("Products", { ProductID: oProduct.ProductID }); MessageBox.confirm("Delete " + oProduct.ProductName + "?", { onClose: function (sAction) { if (sAction !== MessageBox.Action.OK) { return; } oModel.remove(sPath, { success: function () { MessageToast.show("Deleted."); } }); } }); }
📡 HTTP
DELETE /odata/v2/InventoryService/Products('P1001')
🔄
Auto-refresh: the V2 model can automatically refresh bindings affected by create/update/remove, so your table updates itself. You can control this with setRefreshAfterChange() — avoid manual refresh(true) after every call, which creates needless requests.
Q: Before calling oModel.remove(), what should you always do?
✓ Ask the user to confirm (e.g. MessageBox.confirm(...)), because deletion may be irreversible. Only call remove() inside the confirmation's callback once they agree.

Model Request Events — Automatic Busy Feedback

The OData model itself fires events when requests start, finish, or fail — perfect for a busy indicator.

onInit: function () { const oModel = this.getView().getModel(); oModel.attachRequestSent( () => this.getView().setBusy(true)); oModel.attachRequestCompleted(() => this.getView().setBusy(false)); oModel.attachRequestFailed( () => MessageBox.error("Request failed.")); }
Q: A user taps a row. How do you get the full record they tapped?
✓ From the event: oEvent.getParameter("listItem").getBindingContext().getObject(). The parameter gives the tapped item, its binding context points to the record, and getObject() returns the full data — ready for an Update or Delete.
Unit 08 · Robustness — Interactive

Errors, Validation & Busy Feedback

Real apps validate before sending, show progress during, and explain clearly after. These three habits separate a toy from a real feature.

1 · Validate Before Any Write

Never send unchecked data to the backend. Guard, then send.

_validateProduct: function (oProduct) { if (!oProduct.ProductID) { MessageBox.warning("Enter the product ID."); return false; } if (oProduct.Price <= 0) { MessageBox.warning("Price must be > 0."); return false; } if (oProduct.Stock < 0) { MessageBox.warning("Stock cannot be negative."); return false; } return true; } // use it: if (!this._validateProduct(oProduct)) { return; }
🛡️
Validate in both places: frontend checks improve the user's experience (instant feedback); backend checks protect the data (the real guard). Never rely on the frontend alone.
2 · Live Validation as the User Types
⚙️ onStockLiveChange
onStockLiveChange: function (oEvent) { const iStock = Number(oEvent.getParameter("value")); const oInput = oEvent.getSource(); if (iStock < 0) { oInput.setValueState("Error"); oInput.setValueStateText("Stock cannot be negative."); } else { oInput.setValueState("None"); } }
3 · Read the Server's Answer — Status Codes

Tap each reply to see what it means. 2xx = worked, 4xx = your request had a problem, 5xx = the server broke.

201 / 204 Success
▸ tap for meaning
It worked. 201 Created after a create; 204 No Content after an update/delete. Your success callback runs — show a toast.
400 / 403 Bad request / Forbidden
▸ tap for cause & fix
You sent something invalid, or lack permission / a CSRF token. For writes, the V2 model normally fetches the security token automatically — a 403 means it wasn't sent or you're not authorized.
404 Not Found
▸ tap for cause & fix
Wrong path or the record doesn't exist. Check the entity set name and the key — this is why createKey() is safer than hand-built paths.
500 Server Error
▸ tap for cause & fix
The backend logic failed — not your UI's fault. The response body usually carries a business message; surface it to the user and report to the backend team.
4 · A Reusable Error Parser
_getErrorMessage: function (oError) { try { const o = JSON.parse(oError.responseText); return o.error?.message?.value || "The operation failed."; } catch (e) { return oError.message || "An unexpected error occurred."; } }
💬
Backend messages are often meaningful — "Product cannot be deleted because orders exist", "Record was changed by another user". Parse and show them so users understand why something failed.
Q: Why validate on the frontend if the backend also validates?
✓ Frontend validation gives instant, friendly feedback and avoids needless round-trips; backend validation is the real protection for business data. You want both — they serve different purposes.
