Unit 01 · Foundation — Animated

REST Principles

REST = Representational State Transfer — an architectural style, not a protocol or technology. It is defined by 6 constraints. Think of it as the traffic law every web API must obey.

The Restaurant Analogy — Watch It Play
You don't walk into the kitchen — you tell the waiter (HTTP) what you want, using a standard way of asking. Press play to watch the round-trip.

The other three constraints — Cacheable, Layered System, and Code on Demand (optional) — round out the six, but they're secondary for tomorrow. Focus your revision on the three Core ones above.

📌
REST is a set of rules, not a technology. Every request must be self-contained (stateless) — your auth token goes in every single call. Resources have URLs, and you act on them with standard HTTP verbs. The client (SAPUI5) and server (Gateway) are fully decoupled.
Q: What does "stateless" mean in REST? Give one example.
✓ Stateless means every request carries all the info the server needs — the server remembers nothing between calls. Example: a SAPUI5 app sends its Authorization: Bearer token on every OData request, not just the first login. This is what lets any server in a cluster handle any request (scalability).

❌ Stateful (non-REST)
Server keeps a session in memory. Only that server knows who you are. Can't freely scale — you're pinned to one machine.
✅ Stateless (REST)
Every call self-contained (token inside). Any server handles any request. This is why SAP hub deployment & load balancing work.
Q: Why does the client read the status code before the body?
✓ The status code tells the client whether the body is even usable. A 200 means "use the data"; a 404/403/500 means the body is an error, not the resource. Branching on the code first avoids trying to render an error payload as if it were valid data.
Unit 03 · Concept

OData Building Blocks

OData = REST + a standard rulebook for business data. If REST is general traffic law, OData is the specific highway code for enterprise apps — standardized filtering, sorting, paging, and navigation.

Mapped to Database Concepts
DatabaseOData EquivalentExample
TableEntity TypeLeaveRequest
All rows in tableEntity SetLeaveRequestSet
One specific rowEntityLeaveRequestSet('LR001')
Column / FieldPropertyEmployeeID, Status
Foreign key / JOINNavigation PropertyLeaveRequest → ToEmployee
Stored procedureFunction ImportApproveLeave()
The Blueprint — $metadata

Every OData service exposes a metadata document — the contract between client and server. SAPUI5 reads it automatically on startup.

GET /sap/opu/odata/sap/ZLEAVE_SRV/$metadata → HTTP 200 <EntityType Name="LeaveRequest"> <Key><PropertyRef Name="LeaveRID"/></Key> <Property Name="LeaveRID" Type="Edm.String"/> <Property Name="EmployeeID" Type="Edm.String"/> <Property Name="Status" Type="Edm.String"/> <NavigationProperty Name="ToEmployee" .../> </EntityType>
🔗
Try a real service: the public OData V4 playground exposes a live $metadata document at services.odata.org/V4/OData/OData.svc/$metadata. Open it and you'll see the same EntityType / Property / NavigationProperty structure — for Products, Categories, and Suppliers. You'll build queries against it in the next unit.
🛠️
Always add ?$format=json — the default response is XML, which SAPUI5 still handles, but JSON is cleaner and lighter. OData wraps results in d.results for a collection, or just d for a single entity.
Q: What is the difference between an Entity Type and an Entity Set?
✓ An Entity Type is the structure/blueprint (like a table definition — fields and their types). An Entity Set is the collection URL you actually query (like all the rows). Example: LeaveRequest is the type; LeaveRequestSet is the set you hit with GET. One record inside is an Entity: LeaveRequestSet('LR001').

Q: Which query option would you use to show only 10 records, skipping the first 20?
✓ Combine $top=10 (LIMIT) with $skip=20 (OFFSET): ?$top=10&$skip=20. This is exactly how server-side paging works in a SAPUI5 table — the grid requests one page at a time instead of pulling every row.

🔐
Anything that changes data (POST/PUT/PATCH/DELETE) needs a CSRF token first. Fetch it with a header X-CSRF-Token: Fetch, then send it back on the write request. Forgetting this is the classic 403 Forbidden.
Q: What is the difference between PUT and PATCH?
✓ PUT replaces the entire record — you must send all fields, or missing ones may be blanked. PATCH updates only the fields you send (partial update). Both need a CSRF token and both return 204 No Content on success.

Press play to see each stage light up. Watch who does what — developer vs Basis admin.
🧩
SEGW generates 4 ABAP classes — MPC, MPC_EXT, DPC, DPC_EXT. You only ever write code in the _EXT (extension) classes; SAP regenerates the base classes. Your GET_ENTITYSET logic lives in the DPC_EXT.
🏠 Embedded Deployment
Gateway lives on the same server as the backend. Skip /IWFND/MAINT_SERVICE. Just SEGW → SICF.
🌐 Hub Deployment
Separate Gateway server. You must run /IWFND/MAINT_SERVICE to tell the hub the service exists on backend system X (via RFC).
Q: Which 3 transactions register an OData service — and in what order?
✓ (1) SEGW — build & generate the service. (2) SICF — activate the URL node so it responds (without this → 404). (3) /IWFND/MAINT_SERVICE — register on the Gateway hub (Hub Deployment only; skip for Embedded). Verify by hitting /$metadata — a 200 means it's live.

🧪
Always test /$metadata first. A 200 confirms the service is registered and reachable. If you get 404, it's a SICF activation or wrong service-name problem — not a code bug. Only chase ABAP dumps in ST22 after you see a 500.
Q: You run a POST in Gateway Client and get HTTP 403. What's wrong?
✓ You forgot the CSRF token. Write operations (POST/PUT/DELETE) require it. Fix: first send a request with header X-CSRF-Token: Fetch (or click "Fetch CSRF Token"), copy the token from the response header, then paste it into your POST request header and re-execute.
Unit 08 · Synthesis

How It All Connects

The four topics stack into one continuous story — from the rules, to the standard, to publishing, to testing, to handing the URL to the frontend.

🧠
The mental model for tomorrow:
1. REST = the rulebook (stateless, uniform, client-server)
2. OData = REST specialised for business data (entity sets + $ options)
3. Registration = SEGW → SICF → /IWFND/MAINT_SERVICE makes the URL live
4. Gateway Client = test /$metadata → GET → $filter → POST before frontend
5. Then → hand the URL to the SAPUI5 developer
